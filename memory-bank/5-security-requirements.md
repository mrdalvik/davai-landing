# Security Requirements: Davai Landing Page

## Threat Model

Davai Landing — статический сайт на GitHub Pages без бэкенда, баз данных и аутентификации. Поверхность атаки минимальна, но не нулевая.

**Что под угрозой:**
- Целостность контента (пользователь должен видеть наш код, а не подменённый)
- Приватность посетителей (утечка данных через сторонние ресурсы)
- Безопасность клиентского кода (XSS через i18n-строки, вставляемые в DOM)

**Потенциальные атакующие:**
- Скрипт-кидди: автоматические сканеры, попытки эксплуатировать типовые уязвимости
- Supply chain: компрометация внешних ресурсов (Google Fonts CDN)
- Контрибьюторы: случайное или намеренное внедрение вредоносного кода через PR

**Влияние инцидента:** Репутационное. Davai — инструмент для разработчиков; любой security-инцидент на лендинге подрывает доверие к фреймворку.

### Классификация данных

| Данные | Чувствительность | Хранение | Защита |
|--------|-----------------|----------|--------|
| Переводы (JSON) | низкая | /locales/*.json в репозитории | Валидация структуры, экранирование при вставке в DOM |
| Языковые настройки пользователя | низкая | localStorage браузера | Нет PII, только код языка |
| Тема оформления | низкая | localStorage браузера | Нет PII, только строка light/dark |
| IP-адрес посетителя | средняя | Логи GitHub Pages (не наши) + Google Fonts | Минимизация внешних запросов (см. ниже) |

## Аутентификация и авторизация

Не применимо. Нет пользовательских аккаунтов, нет бэкенда, нет API. Авторизация не требуется.

## Валидация ввода и экранирование вывода

| Источник данных | Валидация / Защита | Риск при отсутствии |
|-----------------|-------------------|---------------------|
| JSON-переводы → DOM | Использовать `textContent` для вставки строк. Никогда не использовать `innerHTML` для i18n-строк | Stored XSS: если JSON скомпрометирован (через PR или supply chain), вредоносный HTML/JS исполнится у всех посетителей |
| URL-параметры (если используются для языка) | Валидировать против whitelist: `['en','ru','zh','es','fr','de','ja']` | DOM XSS: подстановка произвольного значения в DOM или fetch-запрос |
| localStorage (язык, тема) | Валидировать при чтении: проверять против whitelist допустимых значений | Маловероятная, но возможная инъекция через devtools или другой скрипт |
| Clipboard API (git clone команда) | Копировать фиксированную строку из кода, не из DOM | Маловероятный risk, но хорошая практика — источник истины в JS, не в HTML |

**Ключевое правило:** В модуле `i18n.js` для подстановки строк из JSON использовать ТОЛЬКО `element.textContent = value`. Для случаев, где нужен HTML (например, ссылки внутри перевода), использовать шаблонные функции с явной конструкцией DOM-элементов (`createElement` + `textContent` + `setAttribute`), а не `innerHTML`.

## Инфраструктура и деплой

### HTTPS
GitHub Pages предоставляет HTTPS автоматически с сертификатом Let's Encrypt. Принудительный HTTPS включён по умолчанию. Проверить в настройках репозитория: Settings → Pages → Enforce HTTPS ✓.

### Content Security Policy (через meta-тег)
GitHub Pages не позволяет задавать HTTP-заголовки. Используем `<meta>` тег в `<head>`:

```html
<meta http-equiv="Content-Security-Policy" content="
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  font-src 'self' https://fonts.gstatic.com;
  img-src 'self' data:;
  connect-src 'self';
  frame-src 'none';
  object-src 'none';
  base-uri 'self';
">
```

**Что это даёт:**
- `script-src 'self'` — блокирует выполнение любых скриптов не из нашего домена (и inline-скриптов)
- `connect-src 'self'` — fetch() работает только для наших JSON-файлов
- `frame-src 'none'` — сайт нельзя встроить в iframe (clickjacking protection)
- `object-src 'none'` — блокирует Flash/Java плагины
- `style-src 'unsafe-inline'` — необходим для CSS Custom Properties при смене темы. Если стили темы применяются через CSS-классы на `<body>`, можно убрать `unsafe-inline`

**Угроза:** `'unsafe-inline'` в `style-src` ослабляет CSP. Цель — избавиться от него, применяя тему через CSS-классы, а не inline-стили.

### Дополнительные meta-теги безопасности
```html
<meta name="referrer" content="no-referrer">
```
Предотвращает утечку URL страницы при переходе по внешним ссылкам.

### CORS
Не применимо — нет собственного API. Все fetch-запросы к `/locales/*.json` идут на тот же origin.

### Rate Limiting
Не применимо — GitHub Pages управляет инфраструктурой.

## Зависимости и supply chain

### Прямые зависимости
**Ноль npm-зависимостей** в runtime — это лучшая защита от supply chain атак. Сохранить этот подход.

### Dev-зависимости (CI/CD)
Минификаторы (`html-minifier-terser`, `clean-css-cli`, `terser`) используются только в CI. Риск ниже, чем у runtime-зависимостей, но:

| Угроза | Требование | Реализация |
|--------|-----------|------------|
| Скомпрометированный минификатор | Зафиксировать точные версии | `package-lock.json` в репозитории, `npm ci` в CI вместо `npm install` |
| Зависимости минификаторов | Периодический аудит | `npm audit` в CI пайплайне |

### Внешние ресурсы (Google Fonts)

| Угроза | Требование | Реализация |
|--------|-----------|------------|
| Google Fonts CDN скомпрометирован → вредоносный CSS/шрифт | Рассмотреть self-hosting шрифта | Скачать JetBrains Mono в `/fonts/`, убрать зависимость от Google Fonts |
| Google собирает IP-адреса посетителей через запросы к fonts.googleapis.com | Self-hosting устраняет проблему | Альтернатива: оставить Google Fonts, но задокументировать в privacy notice |

**Рекомендация:** Self-host JetBrains Mono. Это устраняет единственную внешнюю зависимость в runtime, улучшает приватность (GDPR compliance — нет запросов к Google), ускоряет загрузку (нет DNS-resolve + connection к внешнему серверу) и позволяет ужесточить CSP до `style-src 'self'` / `font-src 'self'`.

### Целостность кода в репозитории

| Угроза | Требование | Реализация |
|--------|-----------|------------|
| Вредоносный PR с XSS в JSON-переводах или JS | Code review для всех PR | Branch protection rules: require review, require status checks |
| Прямой push вредоносного кода | Защита ветки | Branch protection на `main`: no direct push, require PR |

## Чеклист безопасности для разработки

Упорядочено по приоритету и моменту внедрения:

- [ ] **i18n: только textContent** — в `i18n.js` использовать `element.textContent` для всех строк. Запретить `innerHTML` для i18n. Если нужен HTML в переводе — конструировать DOM-элементы программно
- [ ] **CSP meta-тег** — добавить `<meta http-equiv="Content-Security-Policy">` в `<head>` с политикой из раздела выше
- [ ] **Referrer policy** — добавить `<meta name="referrer" content="no-referrer">`
- [ ] **Self-host JetBrains Mono** — скачать шрифт в `/fonts/`, подключить через `@font-face`, убрать ссылку на Google Fonts
- [ ] **Тема через CSS-классы** — переключение темы через класс на `<body>` (не inline-стили), чтобы убрать `'unsafe-inline'` из CSP
- [ ] **Валидация localStorage** — при чтении `lang` и `theme` из localStorage проверять против whitelist допустимых значений
- [ ] **package-lock.json в репозитории** — зафиксировать версии dev-зависимостей, использовать `npm ci` в CI
- [ ] **npm audit в CI** — добавить шаг `npm audit --audit-level=moderate` в GitHub Actions
- [ ] **Branch protection** — включить на `main`: require PR, require review, require status checks
- [ ] **HTTPS enforced** — проверить Settings → Pages → Enforce HTTPS включён
